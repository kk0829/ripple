## 1. 项目背景与目标

### 1.1 背景
当前市面上优秀的局域网文件传输工具（如 LocalSend）解决了跨平台、零配置文件互传的问题，但缺乏**传输过程中的即时沟通能力**。用户在发送文件时，往往需要通过微信/飞书等外部工具附加说明（如"这是修改后的v3版本，请查收"），体验割裂。

### 1.2 目标
打造一款**局域网内的通讯与文件传输融合工具**。核心目标如下：
1. **零配置发现**：接入同一局域网即可自动发现设备，无需注册、登录或中心服务器。
2. **聊天与文件融合**：在聊天窗口中直接发送文本、图片和文件，文件传输进度嵌入聊天气泡。
3. **安全可控**：端到端加密（E2EE），传输数据不经过任何第三方节点。
4. **轻量跨平台**：安装包小巧，支持 Windows、macOS、Linux（后期扩展移动端）。

## 2. 核心功能范围

### 2.1 P0 核心功能（MVP）
- 基于 mDNS 的局域网设备自动发现与心跳保活。
- 1v1 即时文本聊天。
- 基于断点续传的文件/文件夹传输。
- 聊天界面直接拖拽/粘贴发送文件和截图。
- AES-256-GCM 端到端加密（移至 Phase 3 实现）。

### 2.2 P1 增强功能
- 本地 SQLite 消息与文件记录持久化。
- 设备信任机制（信任设备免确认接收，非信任设备弹窗确认）。
- 传输队列管理与限速控制。
- 剪贴板跨设备共享。

### 2.3 P2 远期规划
- 多人临时群聊（星型拓扑）。
- 移动端。
- 局域网协同白板。

## 3. 系统架构设计

### 3.1 技术栈选型

| 层级 | 技术选型 | 选型理由 |
| :--- | :--- | :--- |
| **前端 UI** | React + TypeScript | 聊天 UI 组件生态成熟，虚拟列表性能好 |
| **桌面容器** | Tauri 2.0 | 相比 Electron 包体极小（~5MB vs ~150MB），Rust 后端安全性高 |
| **后端核心** | Rust (Tokio 异步运行时) | 高并发网络 IO 性能优异，内存安全，适合处理文件流 |
| **本地存储** | SQLite (via `rusqlite`) | 轻量级，无需启动独立数据库服务 |
| **设备发现** | mDNS (`tauri-plugin-mdns`) | 基于 Tauri 插件封装，底层调用系统原生 Bonjour/Avahi，兼容性好 |
| **日志系统** | `tracing` + 按天轮转文件 | 结构化日志，便于排查局域网通信问题 |

### 3.2 网络拓扑结构
采用**纯 P2P（Peer-to-Peer）星型无中心拓扑**。每个客户端同时作为服务端和客户端运行。

```
          [Device A]
         /     |     \
        /      |      \
  [Device B]--[Device C]--[Device D]
```
*注：任何两台设备之间的通信都是直接建立的，不经过第三方设备中转。*

### 3.3 系统分层架构

```text
┌─────────────────────────────────────────────────┐
```

## 4. 关键技术模块设计

### 4.1 设备发现与保活
- **发现机制**：启动时在局域网广播 `_ripple._tcp.local.` 类型的 mDNS 服务。
- **信息载荷**：包含设备名称、系统类型、TCP 端口、设备指纹。
- **多网卡处理**：枚举所有非回环 IPv4 地址，全部注册到 mDNS 记录中。
- **离线检测**：采用**主动探活**策略。每 5 秒向已知在线设备发送一次 HTTP Ping (`/api/ping`)，连续 3 次未响应则标记为离线，从 UI 移除。

### 4.2 文件传输协议
基于 HTTP 1.1 Chunked Transfer，保证兼容性和调试便利性。

**交互流程：**
1. **Prepare**: 发送方 POST `/api/prepare`，告知文件名、大小、哈希(SHA256)。
2. **Accept/Reject**: 接收方根据信任策略返回同意或拒绝。
3. **Upload**: 发送方将文件分块（Chunk 大小 64KB），通过 POST `/api/upload/{session_id}` 以流式发送。
4. **Finish**: 传输完毕后发送 `/api/finish`，接收方校验 SHA256 完整性并返回最终状态。
5. **Cancel**: 任意一方可随时调用 `POST /api/cancel/{session_id}` 取消传输。接收方清理 `.part` 文件，发送方停止读取文件流并中断 HTTP 连接。取消后双方通过 WebSocket 互相通知，UI 同步更新传输状态为 `cancelled`。

**断点续传设计：**
在 `Prepare` 阶段，接收方检查本地是否存在同名的 `.part` 文件。若存在且初始 SHA256 匹配，则在 `Accept` 响应中携带 `Range: bytes={already_received}-`，发送方从该偏移量继续上传。

### 4.3 聊天通信协议
基于长连接 WebSocket，保证文本消息的低延迟。

**消息帧格式 (JSON)：**
```json
{
  "id": "uuid-v4",
  "type": "text | file_ref | image | system | ack | typing",
  "timestamp": 1718000000,
  "payload": {
    "content": "你好，这是最新版文档",
    "format": "plain | markdown"
  },
  "ref": null 
}
```

**可靠性保证（简易 ACK 机制）：**
- 发送方维护一个发送队列（Pending Queue）。
- 接收方收到消息后，立即回复 `{"type": "ack", "ref": "消息id"}`。
- 发送方若未收到 ACK，重发该消息（最多 3 次）。初始超时 5 秒，后续根据自适应 RTT 估算动态调整（类似 TCP SRTT），避免过早重发加剧网络拥堵。
- 收到 ACK 后从 Pending Queue 移除并写入 SQLite。

### 4.4 聊天与文件的联动设计（核心差异化）
1. **聊天发文件**：用户在输入框粘贴图片或拖拽文件 → 前端拦截 → 触发 Tauri Command → 后端发起 HTTP 文件传输流程 → 同时向 WebSocket 发送一条 `type: "file_ref"` 消息（包含文件基本信息和 session_id） → 对方聊天窗口显示文件卡片，内部嵌套 HTTP 传输进度条。
2. **文件附言**：在文件管理器右键发送文件时，弹出输入框输入留言 → 打包为复合消息发送。

## 5. 数据模型与存储

采用 SQLite 存储于本地用户目录下 (`~/.ripple/data.db`)。

```sql
-- 设备表（记录曾经遇到过的设备）
CREATE TABLE devices (
    fingerprint TEXT PRIMARY KEY, -- 基于硬件信息生成的唯一ID
    name TEXT NOT NULL,
    os_type TEXT,
    last_ip TEXT,
    trust_level INTEGER DEFAULT 0, -- 0: 陌生, 1: 临时信任, 2: 永久信任
    last_seen_at INTEGER
);

-- 聊天消息表
CREATE TABLE messages (
    msg_id TEXT PRIMARY KEY,
    device_fingerprint TEXT NOT NULL,
    direction TEXT NOT NULL, -- 'sent' 或 'received'
    msg_type TEXT NOT NULL,  -- 'text', 'file_ref', 'system'
    content TEXT,
    transfer_id TEXT,        -- 关联 transfers.session_id，仅 file_ref 类型有值
    created_at INTEGER NOT NULL,
    FOREIGN KEY (device_fingerprint) REFERENCES devices(fingerprint),
    FOREIGN KEY (transfer_id) REFERENCES transfers(session_id)
);
CREATE INDEX idx_messages_device_time ON messages(device_fingerprint, created_at);

-- 文件传输记录表
CREATE TABLE transfers (
    session_id TEXT PRIMARY KEY,
    device_fingerprint TEXT NOT NULL,
    direction TEXT NOT NULL, -- 'upload' 或 'download'
    file_name TEXT NOT NULL,
    file_size INTEGER,
    file_hash TEXT,
    save_path TEXT,
    status TEXT DEFAULT 'pending', -- pending, transferring, completed, cancelled, error
    progress REAL DEFAULT 0.0,
    created_at INTEGER,
    updated_at INTEGER
);
```

## 6. 安全加密方案

采用比 LocalSend 更现代的 **X25519 (ECDH) + AES-256-GCM** 组合。

1. **密钥生成**：每次应用启动时，在内存中生成临时 X25519 密钥对（可选持久化以实现"已知联系人"功能）。
2. **密钥交换**：
   - A 想发消息给 B，A 获取 B 的公钥（通过 mDNS 明文广播或首次握手获取）。
   - A 使用自己的私钥和 B 的公钥进行 ECDH 计算，生成共享密钥。
3. **数据加密**：使用共享密钥派生出的 AES-256 密钥加密 WebSocket 消息体和 HTTP 文件流。每个消息/Chunk 使用 12 字节 Nonce，由 **4 字节连接级随机前缀 + 8 字节自增计数器**组成。每次建立新连接（含重连）时重新协商随机前缀，避免 Nonce 在不同连接间重复。
4. **前向保密**：由于采用临时密钥对，关闭应用后密钥销毁，历史截获的数据无法被解密。

## 7. 日志与可观测性

- Rust 侧使用 `tracing` crate 输出结构化日志，按天轮转写入 `~/.ripple/logs/`，保留最近 7 天。
- 前端侧使用 Tauri 的 log 插件将 JS 日志桥接到 Rust 侧统一管理。
- 日志级别：生产环境默认 `INFO`，开发环境 `DEBUG`，可通过设置页面临时切换为 `TRACE` 用于问题排查。
- UI 内置"导出诊断日志"按钮，将最近日志打包为 `.zip`，方便用户反馈问题。

## 8. 边界场景与异常处理

| 场景 | 应对策略 |
| :--- | :--- |
| **AP 隔离（公共WiFi）** | Ping 探活失败后，UI 底部提示"当前网络可能开启了AP隔离，设备间无法互通"。 |
| **防火墙拦截** | 首次启动时尝试监听端口，若失败则在 UI 弹出图文引导，指导用户放行防火墙。 |
| **同一设备多IP（Wi-Fi+有线）** | mDNS 注册所有 IP；建立连接时，并发尝试所有 IP，选择 TCP 握手延迟最低的 IP。 |
| **大文件传输 OOM** | 严禁将整个文件读入内存。Rust 端使用 `tokio::fs::File` 配合 `BufReader` 进行 64KB 分块流式读写。 |
| **WebSocket 断线** | 采用指数退避重连算法（1s, 2s, 4s, 8s... 最大 30s）。重连成功后恢复通信。注：本项目不支持离线消息，断线期间的消息不暂存不补发。 |
| **中文文件名乱码** | HTTP Header 中文件名强制使用 RFC 5987 编码：`Content-Disposition: attachment; filename*=UTF-8''%xx%xx%xx` |
